# SymCascade：云-边-端三层下的符号-神经级联缓存推理调度器

**设计文档 / 学术论文研究方案**
**日期**：2026-06-27
**目标投稿**：TON（主推）/ TPDS（备选），系统工作 + 理论加分项

---

## 1. 背景与定位

### 1.1 问题与动机

在云-边-端三层异构部署中，LLM Agent 推理面临成本-延迟-质量的多目标权衡。现有方案要么全部上云（高延迟高成本），要么全部在端（低质量），级联路由（FrugalGPT、RouteLLM、AutoMix、C3PO 等）虽能缓解，但其"低成本层"均为小 LLM，存在两个未被覆盖的空白：

1. **无人将符号规划器作为级联的低成本确定性推理层**——现有级联/推测式工作（C3PO、ConfSpec、SpecSearch、SpecCoT）的低成本层全部是小 LLM，无一人引入符号规划器（PDDL + Fast Downward）。
2. **无人将语义缓存与符号规划结合**——缓存的不只是最终答案，而是符号规划器产出的 PDDL 骨架，命中时用符号搜索器做受约束重规划。

这两个空白经文献验证（覆盖 9 篇主流级联/路由论文 + 7 篇符号规划论文 + 6 篇推测式推理论文），截至 2026 年 6 月均无对应工作。

### 1.2 一句话定位

在边缘 Gemma 4 12B 小模型层引入一条「语义缓存 → 符号规划 → 同源置信度路由」的三级低成本级联，让大部分请求在抵达云端大模型前就被缓存复用、符号规划或边缘高置信度推理拦截，从而在质量保持的前提下实现「降云端调用 + 降延迟 + 降成本」的多目标 Pareto 改进。

## 2. 核心创新点

### C1：符号规划器作为「缓存未命中后的低成本确定性推理层」

**机制**：缓存未命中时，不直接升级到小 LLM / 大 LLM，而是先尝试符号规划器（PDDL + Fast Downward/LAMA）做低成本确定性求解；不可形式化或求解失败才升级到 LLM。

**文献空白**：C3PO（NeurIPS 2025）、ConfSpec、SpecSearch、SpecCoT 等所有级联/推测式工作的"低成本层"都是小 LLM，无一人用符号规划器。符号规划器在 LLM agent 中的角色均为"求解器后端"（LLM+P）、"验证器"（RNSP）或"reward 源"（SELF-CRITEACH），从未被纳入级联作为低成本层。

**价值**：符号规划器具有"低成本、可验证、确定性"特性——CPU 上运行零 GPU 占用，求解结果可经 VAL 形式化校验，且对可形式化子问题给出最优解。

### C2：缓存 PDDL 骨架 + 符号重规划复用（论文主创新点，做深）

**机制**：语义缓存不只存最终答案，还缓存符号规划器产出的 **PDDL plan skeleton（带前置条件/效果注解的动作序列模板）**。命中时由符号规划器在骨架上做**受约束重规划**（fast replanning / LAMA-style restart from cached prefix），而非用小 LLM 适配。

**文献空白**：
- APC（Agentic Plan Caching, NeurIPS 2025）确实缓存 plan template，但复用机制是**小 LLM 适配**，不是符号搜索重规划。
- LTM-STM（MDPI Symmetry 2024）缓存 PDDL skeleton，但领域特定（电池拆解），用 STM 修改而非符号搜索复用。
- **"语义 embedding 模糊匹配 → 命中 PDDL 骨架 → 符号搜索器受约束重规划"这一明确组合，文献中未见。**

**自研内容（无现成代码）**：
1. PDDL 骨架抽取器——把已解 plan 泛化为可复用 skeleton（动作序列模板 + 谓词模式 + 目标结构）
2. 骨架语义匹配评估器——GPTCache 提供插槽但无 PDDL 专用实现，需自写结构化距离度量（动作序列编辑距离 + 谓词图结构相似度）
3. 受约束重规划器——命中骨架时用 Fast Downward 在骨架前缀上做 fast replanning
4. 命中/重规划决策逻辑——何时复用骨架、何时重解、如何把 skeleton 实例化为具体 problem

### C3：Gemma-Gemini 同源置信度门控

**机制**：利用 Gemma 4 12B 由 Gemini 3 研究/技术构建的同源性——其"自我知识边界"与 Gemini 高度对齐（Gemma 自信的题 Gemini 大概率也能做对，Gemma 不自信的题正是 Gemini 能补的差）。用 vLLM 的 logprobs 输出计算置信度，叠加 MAPIE conformal prediction 给分布无关覆盖率保证，作为级联升级门控。

**文献空白**：现有级联用异构小模型（如 Llama→GPT）置信度，无同源红利；专门针对 Gemma 的置信度校准工作未见发表。Gemma 4 与 Gemini 3 同源（优于 Gemma 3 与 Gemini 2.0 同源），使置信度对齐更可靠。

**注意**：Gemma 4 的 thinking mode 会消耗更多 token，作为低成本层时必须关闭（`enable_thinking=False`），仅在做难度预估时可选开启。

### C4：级联最优性定理拓展（理论加分项，偏 TON/TPDS）

**机制**：把 ICML 2025 Dekoninck 等的级联最优性定理从「质量-成本 2 目标 + 模型池」拓展到「质量-延迟-成本 3 维 Pareto + 含缓存命中短路层 + 符号规划层」，证明贪心按 benefit-cost 比逐层升级的 ε-最优性。

**文献空白**：原定理仅限 2 目标、模型池场景，未含延迟维度、未含缓存层、未含符号规划层、未含云边端异构。HybridFlow 的 knapsack 阈值结构 r*=𝓘[Δq/c>λ] 已暗示贪心结构，但未形式化证明其在含缓存/符号层时的最优性。

**工作**：纯理论推导，无代码。需扎实的证明（可能给出 ε-最优界而非严格最优）。

## 3. 系统架构与数据流

### 3.1 级联管线

```
请求到达
   │
   ▼
[L0: 语义缓存层] ──命中──► 直接返回 (零 LLM 调用)
   │未命中
   ▼
[L1: Gemma 4 12B 置信度预估] (thinking=False, 取 logprobs)
   │
   ├──高置信 (conformal 校准)──► Gemma 直接答 (边缘完成)
   │
   └──低置信
       ▼
   [L2: PDDL 骨架缓存]
       │
       ├──骨架命中──► 受约束重规划 (Fast Downward, CPU)
       │                  │
       │                  └──求解成功──► 返回 + 抽取骨架存缓存
       │
       └──未命中──► [L3: LLM 生成 PDDL → Fast Downward 求解] (C1)
                          │
                          ├──可形式化且求解成功──► 返回 + 抽取骨架存缓存
                          │
                          └──不可形式化/求解失败
                              ▼
                          [L4: 云端 Gemini 3 兜底]
```

### 3.2 各层成本-质量特性

| 层级 | 成本 | 延迟 | 质量 | LLM 调用 | GPU |
|------|------|------|------|---------|-----|
| L0 语义缓存 | ~0 | ~0 (内存查询) | 同缓存原值 | 0 | 无 |
| L1 Gemma 边缘 | 0 (本地) | 低 | 中-高(置信度高时) | 1 | 边缘 GPU |
| L2 骨架重规划 | 低 | 低 (CPU 搜索) | 高 (确定性最优) | 0 | 无(CPU) |
| L3 LLM+符号规划 | 中 | 中 | 高 | 1 (生成 PDDL) | 边缘 GPU + CPU |
| L4 云端 Gemini | 高 | 高 | 最高 | 1 | 云端 |

### 3.3 数据流关键约束

- **置信度门控阈值**：由 MAPIE conformal prediction 在校准集上学习，保证覆盖率 1-α（非手动调参）
- **骨架缓存键**：embedding 由轻量编码器（如 BGE-M3）对 PDDL 骨架结构编码，cosine similarity > θ 命中；θ 自适应
- **升级率约束**：每层升级率须 < 33-66%，否则级联不划算（需实验验证）
- **目标级联深度**：平均深度 < 1.5（多数请求在 L0/L1 即结束）

### 4.4 结果评判方法

补齐现有级联/路由论文的显著性检验短板（9 篇中仅 3 篇做了形式化处理）。

**第一层：客观指标优先（避免 LLM-as-judge 噪声）**
- ALFWorld/WebArena：任务成功率（程序化检查，无 judge）
- HotpotQA/GSM8K：EM/F1（与 ground truth 比对）
- GAIA：exact-match（刻意避免 judge 以保证可复现）
- 仅开放式轨迹质量评估时用 judge：双向交换位置 + 3 judge 投票，避免用被评模型同族做 judge

**第二层：Pareto 多目标评判**
- 超体积 HV：各方法近似前沿的超体积，bootstrap 95% CI 比较支配显著性（参考点取 nadir）
- Pareto 支配判断：本方法前沿在基线左下方即整体支配
- ε-指示子：二元直接比较"本方法用多大 ε 才能 ε-支配基线"
- 呈现：Pareto 前沿图（证支配）+ 采样数值表（证精确）

**第三层：统计显著性检验**
- ≥10 次独立运行（不同随机种子），报告 mean±std
- Wilcoxon signed-rank test（配对非参数，不假设正态，对 Pareto 指标偏态分布稳健）
- Bootstrap 置信区间（B=1000-5000 次重采样，取 2.5%/97.5% 分位）
- 多重比较校正：与 5 个基线对比用 Holm 校正（比 Bonferroni 温和），或 Friedman+Nemenyi 排名检验
- 效应量：报告 Cliff's delta 说明差异实际意义

**级联系统特有指标（E6）**：
- 各层命中率/升级率分布（升级率须 < 33-66%）
- 级联深度直方图（目标平均深度 < 1.5）
- 成本/质量/延迟各层贡献分解（Sankey 图）

### 5.2 按创新点拆解

**C1 符号规划层（改造为主）**
- 起点：`itl-ed/llm-dp`（LLM-DP）已在 ALFWorld 跑通"LLM 生成 PDDL→Fast Downward 求解"，96% 成功率
- 改造：LLM 后端从 GPT-3.5 换成 Gemma 4 12B（vLLM serve），接入 conformal 置信度门控
- 自研：把符号规划器封装成级联的一个 stage（定义 stage 接口：输入 query→输出 result+置信度）
- 工作量：低

**C2 PDDL 骨架缓存+符号重规划（核心自研，做深）**
- 起点：GPTCache 的模块化缓存键插槽（自定义 embedding_func 承载 PDDL 骨架编码）
- 架构启发：`ace-agent/ace` 的 Generator-Reflector-Curator 结构（仅借鉴缓存架构思路，对象为 NL 非 PDDL）
- 必须从 0 写（无现成代码）：
  1. PDDL 骨架抽取器
  2. 骨架语义匹配评估器（GPTCache 插槽无 PDDL 专用实现）
  3. 受约束重规划器
  4. 命中/重规划决策逻辑
- 工作量：中高（论文主创新点）

**C3 同源置信度路由（改造+自研）**
- 起点：AutoMix 的 POMDP verifier + RouteLLM 的 router 基座
- 改造：配置模型对为 Gemma 4→Gemini 3（RouteLLM 支持自定义）
- 自研：C3PO 级联版无公开代码，需基于论文实现同源置信度路由逻辑
- 工作量：中

**C4 理论加分项（纯推导）**
- 拓展 ICML 2025 级联最优性定理
- 工作量：低（理论推导）

### 5.3 许可证兼容性

Fast Downward 是 GPL-3.0，直接链接会触发 GPL 传染性。规避方案：用 `unified-planning`（Apache-2.0）作中间层，通过 subprocess 调用 FD（进程隔离，不触发传染）。其余起点均为 MIT/Apache/BSD，兼容性好。Gemma 4 12B 为 Apache-2.0（优于 Gemma 3 的 Gemma Terms）。

## 7. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 符号规划器对非 ALFWorld 任务形式化困难 | C1/C2 适用范围受限 | ALFWorld 为主载体；HotpotQA/GAIA 上仅验证 C3；明确论文边界为"可形式化子问题" |
| Gemma 4 刚发布 24 天，生态可能不完善 | 部署受阻 | vLLM 已有 Usage Guide；备选 Gemma 3 12B/27B（生态成熟，同源 Gemini 2.0） |
| 骨架缓存命中率低 | C2 价值不显著 | 预实验在 ALFWorld 6 类任务上测命中率；若低则扩展至 Blocksworld/Gripper 纯 PDDL 集保证命中 |
| 级联多次尝试累积延迟 | 端到端延迟反而增加 | 置信度门控早停 + 缓存命中短路；E3 专门验证延迟分布 |
| 思考模式 token 消耗 | 低成本层变贵 | 低成本层强制 `enable_thinking=False`，设计文档已明确约束 |
| C4 理论证明困难 | 理论加分项落空 | 降级为 ε-最优界而非严格最优；若仍困难则改为实验性验证贪心最优性 |

## 8. 决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 方向 | 融合多个 Idea (#3+#10+#12) | 三机制咬合产生单一机制无法实现的新贡献 |
| 价值指标 | 多目标联合(延迟-质量-成本 Pareto) | 学术故事性最强 |
| 部署场景 | 云-边-端三层 | 与 #3/#10/#12 天然契合 |
| 实验条件 | 仿真为主 | 已明确，级联每级可独立消融 |
| 模型选型 | Gemma 4 12B | 与 Gemini 3 同源(C3)、性能远超 Gemma 3、Apache 2.0、vLLM 已支持 |
| C2 深度 | 做深(4 周自研) | 论文主创新点，文献空白最清晰 |
| C4 定位 | 理论加分项(偏 TON/TPDS) | 不喧宾夺主，作为系统工作的理论支撑 |
| 代码起点 | 基于 LLM-DP 改造 | 已 96% 成功率，增量是降 LLM 调用而非提准确率 |
| GPL 规避 | unified-planning 进程隔离 | 不触发 GPL 传染性 |
